# Supabase Schema v1 Draft

This is a draft Supabase/Postgres schema plan for a future production backend for Young's Learners. It converts the migration and auth/onboarding blueprints into table-level planning. It is documentation only: do not create SQL migrations, connect Supabase, or use real student, parent, teacher, school, fee, payment, or homework data until the security foundation is complete.

## 1. Schema overview

Young's Learners should use Supabase Auth for identity, Supabase Postgres for operational records, Supabase Row Level Security for access control, and Supabase Storage for private files. The schema should support one shared platform with role-specific access for:

- `hq_admin`
- `branch_supervisor`
- `teacher`
- `parent`
- `student`

Frontend navigation and filtering are user experience tools only. The database must enforce access.

## 2. Core identity tables

### `profiles`

Purpose: one app profile per Supabase Auth user.

Key fields:

- `id uuid primary key references auth.users(id)`
- `email text`
- `phone text`
- `full_name text not null`
- `role user_role not null`
- `branch_id uuid null references branches(id)`
- `account_status account_status not null default 'active'`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

Relationships:

- May link to `guardians`.
- May link to `teachers`.
- May create `staff_invites`, `teacher_tasks`, `task_attachments`, and audit records.

Role access rules:

- Users can read their own profile.
- HQ Admin can read/manage all profiles.
- Branch Supervisor can read staff profiles in their branch.
- Teachers, parents, and students cannot browse all profiles.

### `branches`

Purpose: education centre branch records.

Key fields:

- `id uuid primary key`
- `name text not null`
- `code text unique`
- `address text`
- `phone text`
- `account_status account_status not null default 'active'`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

Relationships:

- Has many `classes`, `students`, `profiles`, `trial_schedules`, `fee_records`, and `observations`.

Role access rules:

- HQ Admin can manage all branches.
- Branch Supervisor can read own branch.
- Teacher can read branch for assigned classes.
- Parent/student can see limited branch context for linked/self records only.

### `staff_invites`

Purpose: first-time staff onboarding codes.

Key fields:

- `id uuid primary key`
- `email text not null`
- `role user_role not null`
- `branch_id uuid references branches(id)`
- `class_id uuid null references classes(id)`
- `code_hash text not null`
- `expires_at timestamptz not null`
- `used_at timestamptz null`
- `used_by uuid null references profiles(id)`
- `generated_by uuid references profiles(id)`
- `created_at timestamptz not null default now()`

Relationships:

- Generated by HQ Admin or Branch Supervisor.
- Can pre-fill `profiles`, `teachers`, or `teacher_class_assignments`.

Role access rules:

- HQ Admin can manage all invites.
- Branch Supervisor can manage invites for own branch.
- Staff can validate only their own invite through a server-side flow.

### `onboarding_sessions`

Purpose: temporary onboarding progress state.

Key fields:

- `id uuid primary key`
- `profile_id uuid null references profiles(id)`
- `onboarding_type text not null`
- `status text not null`
- `draft_payload jsonb`
- `expires_at timestamptz`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

Relationships:

- May later create or update `profiles`, `guardians`, `students`, `student_school_profiles`, and `teacher_class_assignments`.

Role access rules:

- User can access own onboarding session.
- Admin staff can access sessions they are permitted to supervise.
- No public browsing.

## 3. Student and family tables

### `students`

Purpose: learner identity and enrolment record.

Key fields:

- `id uuid primary key`
- `branch_id uuid references branches(id)`
- `class_id uuid references classes(id)`
- `full_name text not null`
- `date_of_birth date null`
- `student_number text unique`
- `account_status account_status not null default 'active'`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

Relationships:

- Belongs to a branch and current class.
- Links to guardians through `guardian_student_links`.
- Has `student_school_profiles`, `student_learning_profiles`, attendance, homework, reports, fees, and reminders.

Role access rules:

- HQ Admin can access all students.
- Branch Supervisor can access own branch students.
- Teacher can access students in assigned classes.
- Parent can access linked children only.
- Student can access own record only.

### `guardians`

Purpose: parent/guardian profile extension.

Key fields:

- `id uuid primary key`
- `profile_id uuid references profiles(id)`
- `contact_email text`
- `contact_phone text`
- `verification_status text not null default 'pending'`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

Relationships:

- Belongs to one profile.
- Links to students through `guardian_student_links`.

Role access rules:

- Guardian can read own record.
- HQ Admin can manage all guardians.
- Branch Supervisor can read guardians linked to own branch students.

### `guardian_student_links`

Purpose: secure parent-child access relationship.

Key fields:

- `id uuid primary key`
- `guardian_id uuid references guardians(id)`
- `student_id uuid references students(id)`
- `relationship text`
- `is_primary boolean not null default false`
- `account_status account_status not null default 'active'`
- `created_at timestamptz not null default now()`

Relationships:

- Joins `guardians` to `students`.
- Drives parent RLS for student records, reports, homework, attendance, and fees.

Role access rules:

- Parent can read own active links.
- HQ Admin can manage all links.
- Branch Supervisor can manage links for own branch students.

### `schools`

Purpose: school directory and context.

Key fields:

- `id uuid primary key`
- `name text not null`
- `school_type school_type`
- `country text`
- `region text`
- `account_status account_status not null default 'active'`

Relationships:

- Has many `student_school_profiles`.
- May map to likely `curriculum_pathways`.

Role access rules:

- Non-sensitive school catalog data may be readable by authenticated users.
- Student-specific school links remain private through `student_school_profiles`.

### `curriculum_pathways`

Purpose: syllabus/pathway catalog.

Key fields:

- `id uuid primary key`
- `name text not null`
- `curriculum_type curriculum_type`
- `school_type school_type`
- `subject text`
- `level text`
- `description text`
- `account_status account_status not null default 'active'`

Relationships:

- Linked by `student_school_profiles`.
- Later linked to `curriculum_knowledge_base`, AI marking, and revision resources.

Role access rules:

- Catalog can be readable by authenticated users.
- Student pathway assignment is private and scoped to student access.

### `student_school_profiles`

Purpose: learner-specific school and syllabus context.

Key fields:

- `id uuid primary key`
- `student_id uuid references students(id)`
- `school_id uuid references schools(id)`
- `year_grade text`
- `school_type school_type`
- `curriculum_pathway_id uuid references curriculum_pathways(id)`
- `subjects_enrolled text[]`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

Relationships:

- Belongs to one student.
- References school and curriculum pathway.
- Supports Phase 2 learning personalisation.

Role access rules:

- Same as student access.
- Parent can read linked child profile.
- Teacher can read assigned student context.

### `student_learning_profiles`

Purpose: learning preferences, level, and personalisation metadata.

Key fields:

- `id uuid primary key`
- `student_id uuid references students(id)`
- `learning_level text`
- `strengths text[]`
- `focus_areas text[]`
- `gamification_preferences jsonb`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

Relationships:

- Belongs to one student.
- Later informs learning resources, gamification, and AI recommendations.

Role access rules:

- Same as student access.
- Teachers can read/update for assigned students if allowed by product policy.
- Parents/students can read approved student-facing information only.

## 4. Class and teacher tables

Purpose: class records linked to branches and students.

Key fields:

- `id uuid primary key`
- `branch_id uuid references branches(id)`
- `name text not null`
- `subject text`
- `level text`
- `schedule text`
- `account_status account_status not null default 'active'`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

Relationships:

- Belongs to one branch.
- Has many students.
- Has many teacher assignments.

Role access rules:

- HQ Admin can access all classes.
- Branch Supervisor can access own branch classes.
- Teacher can access assigned classes.
- Parent/student can read class summary for linked/self student only.

### `teachers`

Purpose: teacher profile extension.

Key fields:

- `id uuid primary key`
- `profile_id uuid references profiles(id)`
- `branch_id uuid references branches(id)`
- `staff_code text unique`

- `created_at timestamptz not null default now()`

Relationships:

- Belongs to `profiles`.
- Has many `teacher_class_assignments`.
- Has many `teacher_tasks`.

Role access rules:

- Teacher can read own teacher record.
- Branch Supervisor can access own branch teachers.
- HQ Admin can access all.

### `teacher_class_assignments`

Purpose: class-level access and teaching assignment.

Key fields:

- `id uuid primary key`
- `teacher_id uuid references teachers(id)`
- `class_id uuid references classes(id)`
- `branch_id uuid references branches(id)`
- `assignment_role text`
- `account_status account_status not null default 'active'`
- `created_at timestamptz not null default now()`

Relationships:

- Joins teachers to classes.
- Drives teacher RLS for classes, students, attendance, reports, homework, and tasks.

Role access rules:

- Teacher can read own active assignments.
- Branch Supervisor can manage own branch assignments.
- HQ Admin can manage all assignments.

## 6. Future AI learning tables

These tables are Phase 2+ and should not be implemented before the MVP backend, auth, RLS, and storage policies are stable.

Key fields: `id uuid`, `homework_attachment_id uuid`, `student_id uuid`, `extraction_status text`, `extracted_question_text text`, `extracted_answer_text text`, `created_at timestamptz`.

Access rules: same as homework attachment access, with teacher review required before parent/student visibility.

Access rules: internal staff/system access; public exposure only through curated learning materials.

### `teacher_follow_up_tasks`

Purpose: AI- or supervisor-generated follow-up actions.

Key fields: `id uuid`, `student_id uuid`, `class_id uuid`, `teacher_id uuid`, `source text`, `action_title text`, `action_detail text`, `task_status task_status`, `created_at timestamptz`.

Access rules: same as teacher tasks; parent/student visibility only after approval.

### `student_reminders`

Purpose: student-facing homework, revision, and habit reminders.

Key fields: `id uuid`, `student_id uuid`, `reminder_type text`, `title text`, `delivery_channel text`, `cadence text`, `task_status task_status`, `created_at timestamptz`.

Access rules: student/linked parent can read own reminders; staff scoped by class/branch.

## 8. RLS policy plan

Frontend filtering is not security. Database RLS must enforce access for every table containing private operational, learner, family, staff, fee, report, task, or file metadata.

`demoRole` is prototype-only and must never be trusted for production authorization.

Role rules:

- HQ Admin can access all branches and operational records.
- Branch Supervisor can access only records where `branch_id` matches their profile branch.
- Teacher can access assigned classes, students in assigned classes, assigned tasks, and related attendance/homework/reports.
- Parent can access linked children only through `guardian_student_links`.
- Student can access own learning records only.

Recommended helper functions:

- `current_profile()`
- `current_role()`
- `current_branch_id()`
- `is_hq_admin()`
- `can_access_branch(branch_id)`
- `can_access_class(class_id)`
- `can_access_student(student_id)`
- `can_access_teacher_task(task_id)`

## 9. Storage buckets

All buckets should be private. Storage policies must check the database metadata row before allowing read/write access.

### `homework`

Purpose: homework uploads and future scans.

Access rules:

- Parent/student can upload for linked/self student only.
- Teacher can read/review assigned class files.
- Branch Supervisor own branch.
- HQ Admin all.

### `receipts`

Purpose: fee receipt uploads.

Access rules:

- Parent may upload receipt for linked child if product policy allows.
- Branch Supervisor/HQ can review.
- Teacher/student cannot access financial files.

### `task_attachments`

Purpose: lesson plans, materials, and task files.

Access rules:

- Assigned teacher can read task files.
- Supervisor/HQ can manage according to branch scope.

### `observation_files`

Purpose: observation evidence, coaching files, or review attachments.

Access rules:

- HQ Admin all.
- Branch Supervisor own branch.
- Teacher access only when explicitly allowed for completed/self observations.

### `learning_materials`

Purpose: curated student-facing resources.

Access rules:

- Staff can manage according to role.
- Parent/student can read only materials released to their linked/self student or class.

## 11. Safety warning

Do not use real student, parent, payment, homework, school, teacher, fee, or operational data until all of the following are complete:

- Supabase Auth is implemented and tested.
- RLS policies are implemented and tested for every role.
- Storage policies are implemented and tested.
- Backup/export process is ready.
- Role-based QA is complete for HQ Admin, Branch Supervisor, Teacher, Parent, and Student.
- Demo data and real data are clearly separated.

Do not implement this schema yet. Do not create real SQL migration files yet unless explicitly asked.
